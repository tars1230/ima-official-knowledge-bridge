# 发布建议

本项目更适合小范围、精准分发，而不是泛流量式推广。

原因：

- 解决的是明确但相对垂直的痛点
- 对目标用户价值很高，但目标人群并不大
- 运行前提较多，不适合被包装成“开箱即用的大众工具”

## 适合触达的人群

- 正在把 IMA 接入 OpenClaw、Codex、Claude Code 或其他 agent 的用户
- 把 IMA 当作个人知识中枢、外置大脑、参谋系统的重度用户
- 对“官方知识库页面自动化调用”本身有兴趣的开发者
- 维护中文 AI workflow、second brain、knowledge automation 的实践者

## 不建议优先触达的人群

- 只想体验聊天能力的普通 AI 用户
- 没有本地 IMA 环境、也不愿处理本地登录态的人
- 期待 SaaS 化、云端 API 化、零配置接入的人

## 建议的发布顺序

### 第一阶段：验证型发布

目标：判断是否真的有人有同类需求，以及他们最卡在哪里。

建议渠道：

- OpenClaw / skill 社区
- IMA 深度用户群
- 中文 agent / automation 小圈子
- GitHub Discussions、相关仓库 issue、经验帖评论区

建议动作：

- 先发 1 条短介绍
- 附 GitHub 链接
- 明确说明这是面向 advanced users 的 bridge
- 主动写清当前限制，避免吸引错误预期

### 第二阶段：证据型发布

前提：已经补齐至少 1 条真实 live 验证记录。

目标：提高可信度，让外部读者知道这不是停留在概念层。

建议动作：

- 补充一条真实验证记录
- 增加一段最短 demo
- 再做第二轮分发

### 第三阶段：维护型发布

前提：已经出现外部使用者，且反馈指向同一类问题。

目标：决定是否继续投入维护成本。

建议动作：

- 根据真实反馈收敛安装方式
- 判断是否需要补故障排查
- 判断是否值得继续做 discovery → `shareId` 自动提取

## 建议使用的话术风格

- 讲清楚“这个 bridge 解决什么”
- 讲清楚“它不解决什么”
- 不夸张，不包装成万能 IMA 自动化
- 不承诺跨版本无维护成本
- 不承诺无需本地登录态

更适合的定位：

- IMA 官方知识库页面桥接方案
- 面向 agent 的知识库问答接入能力
- 让同会话追问与知识库直问更稳的基础设施

不适合的定位：

- 全自动 IMA 超级插件
- 通用 IMA 逆向平台
- 无门槛知识库 SaaS

## 可直接复用的短介绍

### GitHub 仓库简介版

一个面向腾讯 IMA 桌面端官方知识库页面的程序化 bridge。  
目标不是做 GUI 自动化，而是尽量复用官方知识库页面内部的会话与问答链路，让外部 agent 能更稳定地调用知识库问答、同会话追问，以及已知 `shareId` 的共享知识库试问。

### 社群发帖短版

做了一个 IMA 官方知识库 bridge，重点不是点按 UI，而是直接接官方知识库页面内部的会话链路。  
现在已经支持：

- 个人知识库直问
- 同会话追问复用
- 已知 `shareId` 的共享知识库试问
- 候选知识库巡检

更适合已经在折腾 agent + IMA 工作流的人。  
仓库地址：<https://github.com/tars1230/ima-official-knowledge-bridge>

### 社群发帖长版

最近把 IMA 官方知识库这条链路做成了一个可程序化调用的 bridge，主要是给 agent 用的。

这个项目的重点不是 GUI 自动化，也不是拿 OpenAPI 假装等价替代官方知识库问答，而是尽量贴近 IMA 官方知识库页面本身的会话 / presenter / session 链路。

当前能做的事情包括：

- 对个人知识库直接提问
- 在同一个知识库会话里连续追问
- 已知 `shareId` 时，不加入也能先试问共享知识库
- 用 OpenAPI 辅助巡检当前可达的知识库候选

当前限制也比较明确：

- 仍依赖本地 IMA 登录态
- discovery 市场里的 `shareId` 自动提取还没有完全做完
- 不承诺未来所有扩展版本都自动兼容

如果你也在做 IMA + OpenClaw / Codex / 其他 agent 的知识库工作流，这个仓库应该会比较有用。  
仓库地址：<https://github.com/tars1230/ima-official-knowledge-bridge>

## 是否值得持续推广

判断标准不要看 star，而要看这三个信号：

- 是否出现真实复用者
- 他们是否重复提到同一类痛点
- 这些痛点是否值得继续投入维护成本

如果没有出现真实使用者：

- 保持公开即可
- 不必继续投入推广成本

如果出现少量但高质量使用者：

- 值得继续维护
- 值得补 live 记录、demo、排障说明

如果反馈大多是“想开箱即用、想云端托管、想跨平台”：

- 说明需求方向已经偏离当前项目定位
- 不建议为了迎合而把仓库做重
