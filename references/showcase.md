# 30 秒看懂这个项目

这个项目的核心，不是“自动点按 IMA UI”，而是把 **IMA 官方知识库页面内部的真实问答链路** 接成一个可被外部 agent 调用的 bridge。

它解决的问题可以压缩成一句话：

> 当你已经把资料、文档、方法论沉淀在 IMA 知识库里时，外部 agent 不必再退化成坐标点击或重复整理资料，而是可以直接调用官方知识库问答能力。

## 它到底能做什么

### 场景 1：对个人知识库直接提问

输入：

- 一个真实问题
- 一个 `knowledgeBaseId`

输出：

- IMA 官方知识库返回的答案
- 可继续复用的 `sessionId`

这意味着：

- agent 可以把 IMA 当作外部知识中枢
- 问答不再局限于人工打开界面后手动提问

### 场景 2：沿用同一个会话继续追问

输入：

- 新问题
- 上一次返回的 `sessionId`

输出：

- 基于同一知识库会话继续回答

这意味着：

- agent 可以围绕同一个主题连续深挖
- 不必每个子问题都新开一个会话

### 场景 3：先试问共享知识库，再决定要不要纳入

输入：

- 一个真实问题
- 一个已知 `shareId`

输出：

- 对共享知识库的试问结果

这意味着：

- 外部 agent 可以先验证共享知识库是否值得用
- 不必先加入，再发现质量一般

### 场景 4：先巡检候选知识库

输入：

- query
- mode

输出：

- 当前可达的 owned / addable 知识库候选

这意味着：

- agent 可以把“先找候选，再决定问谁”变成一个流程

## 它最适合谁

- 已经在 IMA 里沉淀了很多资料的人
- 想让 OpenClaw / Codex / 其他 agent 调用 IMA 知识库的人
- 不想再依赖 GUI 坐标点击的人
- 想把 IMA 变成外置大脑而不是孤立 App 的人

## 它不适合谁

- 期待零配置、云端托管、纯 SaaS 接入的人
- 只想把 IMA 当普通聊天工具用的人
- 不愿处理本地登录态和版本兼容性的人

## 和普通 GUI 自动化有什么不同

普通 GUI 自动化通常依赖：

- 坐标
- 按钮位置
- 窗口焦点
- 剪贴板

这个项目尽量依赖的是：

- 官方知识库页面内部会话链路
- presenter / session / store 上下文
- 官方页面真实更新流

区别在于：

- 更接近真实产品行为
- 更适合复用会话
- 更适合被 agent 稳定编排

## 一个最小工作流长什么样

1. agent 先判断当前问题应不应该走 IMA 知识库
2. 如果已有明确知识库，直接调用官方知识库问答
3. 如果是连续追问，复用已有 `sessionId`
4. 如果是共享知识库探索，先试问再决定是否继续纳入
5. 如果知识库还不明确，先巡检候选，再决定下一跳

## 这个项目真正有价值的地方

它不是一个“让 IMA 更方便点”的小脚本。

它更像一个桥：

- 一边连着 IMA 里已经沉淀好的高质量资料
- 一边连着外部 agent 的任务拆解、追问、比较、整理能力

一旦这条桥稳定，IMA 就不只是一个需要手动打开的知识库 App，而可以开始变成 agent 工作流中的可调用基础设施。
